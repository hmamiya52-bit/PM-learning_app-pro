# ITサービスマネージャ 頻出テーマ調査仕様 v1.0

作成日: 2026-06-01  
目的: 少ない勉強時間で合格へ近づくため、直近10回分の出題傾向を頻出テーマとして可視化する。

## 1. 調査の位置付け

初回アプリ投入の過去問は直近1年分に限定する。
一方で、知識ノート、頻出テーマ、午後Ⅰ解説、午後Ⅱ参考答案は直近10回分の調査を根拠に設計する。

つまり、アプリ内の問題量は少なくても、学習順序と重点テーマは10回分の傾向から決める。

## 2. 調査対象

対象:

- ITサービスマネージャ試験
- 直近10回分
- 午前Ⅱ
- 午後Ⅰ
- 午後Ⅱ
- 解答例
- 出題趣旨
- 採点講評

初回設計時点の起点:

- 令和7年度春期を最新回とする。
- 実装・拡張時はIPA公式ページで最新公開回を再確認する。

参照元:

- IPA 過去問題一覧: https://www.ipa.go.jp/shiken/mondai-kaiotu/index.html
- 令和7年度過去問題: https://www.ipa.go.jp/shiken/mondai-kaiotu/2025r07.html

## 3. 調査観点

### 3.1 午前Ⅱ

記録する項目:

- 年度/回
- 問番号
- 主テーマ
- 副テーマ
- 用語分類
- 正答
- 午後Ⅰ/午後Ⅱへ接続できるか
- 暗記型か、理解型か

分類例:

- インシデント管理
- 問題管理
- 変更管理
- 構成管理
- リリース管理
- SLA/サービスレベル管理
- 可用性管理
- 容量・能力管理
- ITサービス継続管理
- 情報セキュリティ管理
- 供給者管理
- 資産管理
- ファシリティ管理
- サービスデスク
- 継続的改善
- システム監査/法務

### 3.2 午後Ⅰ

記録する項目:

- 年度/回
- 問番号
- 題材
- 主テーマ
- 副テーマ
- 問われた能力
- 定量読解の有無
- 関係者合意の有無
- 手順/プロセス改善の有無
- 採点講評で指摘された失点
- 午後Ⅱへ流用できる論述ネタ

問われた能力の分類:

- 本文根拠の抽出
- 役割分担の理解
- SLA/目標値の読み取り
- 影響分析
- 原因分析
- 改善策立案
- 関係者合意
- 委託先/CSP管理
- 訓練/評価
- 監視/測定

### 3.3 午後Ⅱ

記録する項目:

- 年度/回
- 問番号
- テーマ
- 主テーマ
- 副テーマ
- 求められる経験
- 設問アの要求
- 設問イの要求
- 設問ウの要求
- 採点講評の注意点
- インフラ案件化しやすいか
- 参考答案の軸

採点講評の分類:

- サービスマネジメント活動ではなくシステム開発活動に寄っている
- 具体性が不足している
- 顧客/利用者の期待分析が不足している
- 改善計画が対策列挙になっている
- CSP/供給者との協議が不足している
- 評価・効果測定が不足している
- 役割や責任分界が曖昧

## 4. テーマ統合ルール

同じテーマを複数名称で扱わない。
午後の解答に使える粒度で統合する。

統合例:

| 生データ上の語 | 統合テーマ |
|---|---|
| インシデント対応、サービスデスク、回避策、障害復旧 | インシデント管理と問題管理 |
| 根本原因分析、既知の誤り、再発防止 | インシデント管理と問題管理 |
| SLA、SLM、サービス報告、顧客満足 | SLA・顧客満足・サービス報告 |
| 可用性、SRE、EB、SLO、稼働率 | 可用性管理・SRE・エラーバジェット |
| BCP、DR、RTO、RPO、バックアップ | ITサービス継続管理・BCP・クラウド活用 |
| 変更、リリース、展開、CAB | 変更管理・リリース管理 |
| CSP、外部委託、供給者、責任分界 | 供給者管理・クラウドサービス活用 |
| CMDB、構成品目、資産、ライセンス | 構成管理・資産管理 |
| アクセス管理、ログ、脆弱性、リスク | 情報セキュリティ・リスク管理 |

## 5. 頻出度ランク

### 5.1 ランクS

条件の目安:

- 直近10回で複数回出題
- 午前Ⅱ、午後Ⅰ、午後Ⅱのうち2区分以上で使える
- 午後答案で行動・判断・改善として書きやすい
- 採点講評でも繰り返し注意される

例:

- インシデント管理と問題管理
- SLA・顧客満足・サービス報告
- 可用性管理・SRE・エラーバジェット
- ITサービス継続管理・BCP・クラウド活用
- 変更管理・リリース管理

### 5.2 ランクA

条件の目安:

- 直近10回で一定頻度あり
- 午前Ⅱまたは午後Ⅰで重要
- 午後Ⅱの補助テーマとして使える

例:

- 供給者管理・クラウドサービス活用
- 構成管理・資産管理
- 情報セキュリティ・リスク管理
- 容量・能力管理
- サービスデスク

### 5.3 ランクB

条件の目安:

- 頻度はS/Aより低い
- 午前Ⅱの暗記として必要
- 午後では補助的に効く

例:

- ファシリティ管理
- 会計/予算管理
- 法務/監査
- サービスカタログ

## 7. アプリ反映ルール

### 7.1 頻出テーマ画面

調査表から以下を反映する。

- rank
- frequency
- years
- appearsIn
- summary
- mustKnow
- morningPattern
- afternoonPattern
- essayPattern
- infraExamples

### 7.2 知識ノート

ランクS:

- 章を必ず作る。
- 午前Ⅱ、午後Ⅰ、午後Ⅱすべての使い方を書く。
- ミニ確認を入れる。

ランクA:

- 章を作る。
- 午後への接続が弱い場合も、インフラ案件例を1つ以上入れる。

ランクB:

- 初回は頻出テーマ画面の補足に留めてもよい。
- 5年分拡張時に章化を判断する。

### 7.3 午後Ⅰ解説

各問題に以下を必須とする。

- purpose
- answerItems
- commentary
- traps
- themeIds

`commentary` は採点講評の長文引用ではなく、学習上の注意に要約する。

### 7.4 午後Ⅱ参考答案

各テーマ2本を基本とする。

2本の差別化:

- 事例が異なる
- 改善軸が異なる
- 利害関係者が異なる
- 評価指標が異なる

例:

| 問題 | 参考答案A | 参考答案B |
|---|---|---|
| 顧客満足 | 監視運用サービス | 社内認証基盤 |
| クラウド活用 | マネージドDB障害 | クラウドバックアップ |

## 8. 初回テーマ

`07_10round_research_matrix.md` の調査結果に基づき、初回テーマは次を正とする。

ランクS:

1. インシデント管理・問題管理・サービスデスク
2. SLA・サービスレベル・顧客満足・事業関係管理
3. 変更管理・リリース/展開管理・構成管理
4. 可用性・サービス継続・容量/能力・SRE

ランクA:

1. 外部供給者管理・クラウド/CSP活用
2. 監視・自動化・データ分析・継続的改善
3. セキュリティ管理・訓練・インシデント対応
4. 資産管理・ファシリティ・運用対象管理

ランクB:

1. 費用・予算・会計・費用最適化
2. 要員育成・チーム改善・動機付け
3. DX・新サービス導入

5年分拡張時には、午前Ⅱの問番号単位分類を追加した上で再集計する。

## 9. 調査完了条件

10回分調査が完了したと言える条件:

- 年度別サマリー表が埋まっている。完了: `07_10round_research_matrix.md`
- テーマ別集計表が埋まっている。完了: `07_10round_research_matrix.md`
- 午後Ⅱテーマ分析表が埋まっている。完了: `07_10round_research_matrix.md`
- ランクS/A/Bの根拠が説明できる。完了: `07_10round_research_matrix.md`
- 知識ノートの章立てに反映されている。実装時に `smKnowledgeSections` へ反映する。
- R7問題解説に関連テーマが紐付いている。実装時に `themeIds` へ反映する。
- 5年分拡張の優先順位が決まっている。完了: `07_10round_research_matrix.md`

## 10. 品質レビュー観点

レビュー時に見ること:

- 単なる午前Ⅱ頻出用語ランキングになっていないか。
- 午後Ⅰ/午後Ⅱで使えるテーマに統合されているか。
- 採点講評の反復指摘が反映されているか。
- インフラエンジニア案件に変換しやすいか。
- 50時間以内に重点学習できる順序になっているか。
- ランクSが多すぎないか。
- ランクBまで初回に作り込みすぎていないか。
