# ITサービスマネージャ寄り道モード 基本設計 v1.0

作成日: 2026-06-01  
前提: `00_requirements.md` に従う  
重要制約: 既存PMドキュメントは改変しない。既存PM機能への影響はトップカードとルート追加に限定する。

## 1. アーキテクチャ方針

ITSMモードはPM Learning App内の独立した機能群として実装する。
トップページから1枚のカードで入り、以降はITSM専用ナビゲーション内で完結する。

設計原則:

1. 既存PM本編に混ぜない。
2. データ、履歴、進捗、保存キーを分離する。
3. 頻出テーマを中心に、午前Ⅱ、午後Ⅰ、午後Ⅱを横断する。
4. 初回は直近1年分を深く作り込む。
5. 5年分拡張に耐えるデータ構造にする。

## 3. 画面責務

### 3.1 ITSMハブ

目的:

- ITSM学習の入口。
- 午前Ⅱ、午後Ⅰ、午後Ⅱの状態を一目で見る。
- 頻出テーマ優先の学習思想を示す。

表示:

- 午前Ⅱ正答率、演習数
- 午後Ⅰ最高点、着手問数
- 午後Ⅱ練習本数、自己評価平均
- 次にやること
- 頻出テーマTop 5
- 知識ノート・専用履歴への導線

判断ロジック:

1. 午前Ⅱ未演習がある場合、午前Ⅱ5問演習を提案。
2. 午前Ⅱが一通り済んで午後Ⅰ未着手がある場合、午後Ⅰ1問演習を提案。
3. 午後Ⅱ練習が少ない場合、骨子または論述練習を提案。
4. それ以外はレポート/履歴確認を提案。

### 3.2 頻出テーマ

目的:

- 「なぜこの順で勉強するのか」を可視化する。
- 午前Ⅱ暗記、午後Ⅰ読解、午後Ⅱ論述を1つのテーマ軸で接続する。

表示:

- 順位
- 頻出度
- 出題年度/回
- 出る試験区分
- 要約
- まず覚えること
- 午前Ⅱでの出方
- 午後Ⅰでの読み方
- 午後Ⅱでの書き方
- インフラ案件例

### 3.3 知識ノート

目的:

- 頻出テーマを短時間で理解する。
- 単なる用語集ではなく、午後答案へ変換できる知識にする。

表示:

- 章一覧
- 頻出度
- 学習目安時間
- 要点
- 午前Ⅱでの使い方
- 午後Ⅰでの使い方
- 午後Ⅱでの使い方
- ミニ確認

### 3.4 午前Ⅱ

目的:

- 令和7年度春期25問を、公式PDF参照で素早く演習する。
- 問題文転載を避け、答案入力と復習に集中する。

表示:

- 公式PDFリンク
- 解答例PDFリンク
- 演習済み数
- 正答率
- フィルタ
- 問番号
- テーマ
- 選択肢ボタン `ア/イ/ウ/エ`
- 正答/直近回答
- 復習メモ

フィルタ:

- すべて
- 未演習
- 直近不正解
- 頻出S

### 3.5 午後Ⅰ

目的:

- 令和7年度春期3問を1問50点で練習する。
- 自分の解答、公式解答例、採点講評、独自ポイントを一画面で見比べる。

表示:

- 問題一覧
- 最新点
- 問題PDF、解答例PDF、採点講評PDF
- 自分の解答メモ
- 公式解答例/要点
- 採点ポイント
- 採点講評からの注意
- 失点しやすい罠
- 点数入力
- 振り返り
- 問題別履歴

### 3.6 午後Ⅱ

目的:

- R7春期2テーマについて、合格答案の型を作る。
- 骨子、本文、自己評価、参考答案を同じ画面で扱う。

表示:

- テーマ一覧
- 問題PDF、出題趣旨PDF、採点講評PDF
- 評価観点
- 骨子例
- 自分の骨子
- 設問ア/イ/ウ本文
- 字数カウント
- 自己評価
- 下書き保存
- 練習完了記録
- 参考答案2本
- テーマ別履歴

### 3.7 ITSM専用履歴

目的:

- ITSM内だけで学習の実績を見る。
- 既存PM履歴には混ぜない。

表示:

- 午前Ⅱ正答率
- 午後Ⅰ記録回数
- 午後Ⅱ練習本数
- 最近のITSMイベント
- 午後Ⅰ記録
- 午後Ⅱ記録
- 午前Ⅱ直近記録

## 4. データ設計

### 4.1 静的データ

配置:

- `src/data/sm/types.ts`
- `src/data/sm/content.ts`

静的データは、まず1ファイルに集約する。
5年分拡張時に以下へ分割する。

- `themes.ts`
- `knowledge.ts`
- `morningQuestions/r7.ts`
- `afternoonProblems/r7.ts`
- `essayProblems/r7.ts`
- `index.ts`

### 4.2 型

主要型:

- `SmFrequentTheme`
- `SmKnowledgeSection`
- `SmMorningQuestion`
- `SmAfternoonProblem`
- `SmEssayProblem`

設計上のポイント:

- すべて `themeId` / `themeIds` を持ち、頻出テーマへ接続できる。
- 問題データは年度拡張に備えて `yearLabel` を持つ。
- 公式資料リンクは `SmSourceLinks` に集約する。
- 午後Ⅱ参考答案は `SmEssaySampleAnswer` として複数保持する。

## 5. 永続化設計

配置:

- `src/lib/sm/progress.ts`

既存PM用 `storage.ts`、`tracker.ts`、`essay.ts`、`morningRecords.ts` は使わない。

### 5.1 保存キー

| キー | 型 | 説明 |
|---|---|---|
| `pmap:sm:morning:records` | `SmMorningRecord[]` | 午前Ⅱの解答履歴 |
| `pmap:sm:afternoon:records` | `SmAfternoonRecord[]` | 午後Ⅰの自己採点履歴 |
| `pmap:sm:essay:attempts` | `SmEssayAttempt[]` | 午後Ⅱの完了済み練習 |
| `pmap:sm:essay:drafts` | `Record<problemId, SmEssayDraft>` | 午後Ⅱ下書き |
| `pmap:sm:events` | `SmHistoryEvent[]` | ITSM内イベント |

### 5.2 方針

- LocalStorageのみ。
- 既存同期対象には含めない。
- 保存失敗時は既存PMへ影響を波及させない。
- イベントは最大100件程度で十分。

## 6. UI設計

### 6.1 共通レイアウト

ITSM内の画面は `SmPageChrome` を共有する。

責務:

- ITSMヘッダー
- PMホームへの戻りリンク
- ITSM内ナビゲーション
- タイトル/説明
- 共通幅と余白

### 6.2 色

PM本編のブランド色とは明確に分けるため、ITSM内はシアン/スレート系を主にする。
ただしアプリ全体のヘッダーや既存レイアウトは変更しない。

色の役割:

- シアン: ITSM内の主要アクション
- スレート: 情報整理、ヘッダー
- ローズ: 頻出S、注意、失点
- アンバー: 採点講評、警告
- エメラルド: 正解/良い状態

### 6.3 レスポンシブ

モバイル:

- ITSM内ナビゲーションは2段グリッドを許容。
- ハブのカードは縦積み。
- 午後Ⅰ/午後Ⅱは左メニューを上部カード一覧へ変える。

デスクトップ:

- 午後Ⅰ/午後Ⅱは左メニュー + 右詳細。
- 頻出テーマは3カラムで観点を並べる。

## 7. コンテンツ方針

### 7.1 公式資料

公式資料はリンクで参照する。
問題文全文は原則アプリ側に持たせない。

保持するもの:

- 問番号
- 正答
- テーマ
- 復習メモ
- 解答例/要点
- 独自解説
- 採点講評の要旨

### 7.2 独自コンテンツ

独自作成:

- 頻出テーマ要約
- 知識ノート
- 午前Ⅱ復習メモ
- 午後Ⅰ採点ポイント
- 午後Ⅰ失点しやすい罠
- 午後Ⅱ骨子
- 午後Ⅱ評価観点
- 午後Ⅱ参考答案

### 7.3 参考答案

参考答案は以下を満たす。

- インフラエンジニア案件。
- 架空事例。
- 実在企業名、実在障害名を使わない。
- サービスマネジメント活動として書く。
- システム開発活動に寄せすぎない。
- 2本の答案は事例・論点を変える。
- 「これは論述例の一つです。唯一の正解ではありません」と表示する。

## 12. 公式情報設計

公式情報の扱いは `05_source_policy.md` に従う。

基本方針:

- 公式問題文全文はアプリ内に保持しない。
- 公式PDFリンクを一次参照として表示する。
- 解答例、出題趣旨、採点講評は学習上必要な要点または独自要約として扱う。
- 各問題データに `source.checkedAt` を持たせる。
- 実装前、レビュー前、5年分拡張前にIPA公式ページを再確認する。

これにより、コンテンツ量を厚くしながら、公式資料の差し替え、制度変更、著作権上の過剰転載リスクを抑える。

## 13. 直近10回分析の反映設計

直近10回の具体的な年度別テーマと頻出ランクは `07_10round_research_matrix.md` を正とする。

アプリ反映:

- ランクSはハブ、頻出テーマ、知識ノート、午後Ⅰ採点ポイント、午後Ⅱ参考答案へ必ず反映する。
- ランクAは知識ノートに章を持たせ、午後Ⅰ・午後Ⅱの補助観点にする。
- ランクBは頻出テーマ画面で補足し、初回では詳細章を増やしすぎない。

初回必須のSランク:

1. インシデント管理・問題管理・サービスデスク
2. SLA・サービスレベル・顧客満足・事業関係管理
3. 変更管理・リリース/展開管理・構成管理
4. 可用性・サービス継続・容量/能力・SRE

初回必須のAランク:

1. 外部供給者管理・クラウド/CSP活用
2. 監視・自動化・データ分析・継続的改善
3. セキュリティ管理・訓練・インシデント対応
4. 資産管理・ファシリティ・運用対象管理

## 14. 50時間ロードマップ設計

学習時間計測は実装しない。

| 枠 | 目安 | 画面 |

| 頻出テーマ把握 | 4h | ハブ、頻出テーマ |
