## 5. 作業手順

### Step 1: 環境セットアップと現状確認
```bash
git pull origin main
git log --oneline -3              # dca4bd1 [C] F1.5-P2 が見えるはず
npm install                        # 念のため
npm run validate-data              # [OK] 全データの整合性確認完了 を確認
npm run build                      # tsc + vite build pass を確認
```

### Step 2: 型整合性チェック（コードレベル）
- `NOTE_DB.stakeholder` 全体を読み、`NoteData` 型と `NoteSection` 型に完全準拠しているか確認
- 主なチェック項目:
  - [ ] `summary: string`（非空）
  - [ ] `sections: NoteSection[]`（33要素）
  - [ ] `exam_tips: string[]`（非空）
  - [ ] 各 `section.heading` が「N. タイトル」形式（連番）で 1〜33 連続
  - [ ] `items` 配列のすべての要素が string
  - [ ] `navyItems` の要素が `EmphasisToken[]`（= `{text, style}[]`） で style は `'navy'` のみ（赤字は item 内 `==X==` で表現する設計）
  - [ ] `headerDiagrams` の各 cell の `bg` が `#` 始まりの hex 文字列
  - [ ] `EmphasisToken.style` が `'red' | 'navy' | 'plain'` 以外を持たない

### Step 3: 構造化トークン整合性
- `==X==` マークアップは半角イコール2つで両端を囲む。次のパターンを検出して報告:
  - [ ] 片側のみの `==`（閉じ忘れ）: `grep -n '==[^=]*$' src/pages/NoteDetail.tsx | grep stakeholder`
  - [ ] 連続 3 つ以上の `=`（例: `===`）
  - [ ] 全角イコール（＝）の混入
  - [ ] `==` 内に改行が含まれる
- `headerDiagrams[].rows[].cells[].label` 内の `\n`（改行）が意図通り（ヘッダ図のみで使用、文中混入なし）

### Step 4: 誤字脱字・表記揺れチェック
- 章番号と heading の整合: 「N. xxx」の N が 1〜33 で重複・欠番なし
- 用語の揺れチェック（以下の用語は表記固定）:
  - [ ] 「ステークホルダー」（「ステイクホルダー」「ステイクホルダ」混入なし）
  - [ ] 「エンゲージメント」（「エンゲイジメント」混入なし）
  - [ ] 「サリエンスモデル」（「セイリエンスモデル」混入なし）
  - [ ] 「マトリクス」と「マトリックス」を統一（本ノートは「マトリクス」採用）
  - [ ] 「コミュニケーション」（カタカナ表記統一）
- 半角/全角の揺れ:
  - [ ] 英数字は半角、括弧は文脈に合わせる（学術用語の英表記は半角丸括弧 `(...)`）
  - [ ] 中黒「・」と読点「、」の使い分けが一貫しているか

### Step 5: コードレベルの整合性
- [ ] `NOTE_DB` の export が維持されているか（line 395 付近）
- [ ] stakeholder 以外のカテゴリが追加されていないか（F1.5 スコープ違反）
- [ ] NoteDetail.tsx の他箇所（NOTE_CATEGORY_IDS など）に意図しない変更が無いか
- [ ] `categories.ts` の stakeholder エントリと heading 整合（カテゴリ名「ステークホルダー」）
- [ ] LocalStorage キーや brand カラー、`pmap:` prefix への影響なし

### Step 6: 検証スクリプト・ビルド再実行
```bash
npm run validate-data
npm run build
```
両方 pass で完了条件達成。

### Step 7: レビュー記録の作成
`tasks/reviews/F1.5-P2_codex_review.md` を新規作成。付録B.3 のテンプレートに従い、観点別に ○/△/✕ で記録。指摘事項は致命／改善推奨に分けて列挙。

### Step 8（任意）: 軽微な修正
- 誤字・表記揺れ・型外しなど **明確な誤り** は同セッション内で修正してよい
- 修正範囲は `NOTE_DB.stakeholder` 内のみ
- コミットは指摘 + 修正をまとめて 1 つの `[X]` コミットでよい
- **コンテンツの中身（PMBOK 解釈・出題傾向の妥当性）に対する変更は禁止** → Gemini/ユーザ判断
