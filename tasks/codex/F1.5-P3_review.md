## 5. 作業手順

### Step 1: 環境セットアップと現状確認
```bash
git pull origin main
git log --oneline -3                # 最新 [C] F1.5-P3 commit を確認
npm install
npm run validate-data               # [OK] + 50問追加が反映されるか確認
npm run build                       # tsc + vite build pass を確認
```

### Step 2: 件数・採番チェック
- [ ] 設問数が **ちょうど50件**
- [ ] id が **重複なく** `pm-st-001` 〜 `pm-st-050` で連続
- [ ] 全 id が `^pm-st-\d{3}$` パターンに準拠
- [ ] 全エントリの `topicId === 'stakeholder'`

### Step 3: 型整合性チェック（Question 型）
各 Question について以下を検証:
- [ ] `id`: string、空でない
- [ ] `topicId`: 'stakeholder' に固定
- [ ] `questionText`: string、`{{blank}}` を **ちょうど1つ** 含む（複数空欄なし）
- [ ] `correctAnswer`: string、空でない
- [ ] `choices`: `string[]` で **長さ4**
- [ ] `choices` に `correctAnswer` が **完全一致で含まれる**（既に validate-data でチェック済だが二重確認）
- [ ] `choices` 内に **重複なし**
- [ ] `explanation`: string、空でない
- [ ] `difficulty`: `1 | 2 | 3` のいずれか
- [ ] `excludeFromWritten` は省略 or `boolean`（typoで他値混入なし）

### Step 4: choices の品質チェック
- [ ] 各設問の誤答3つが **互いに異なる**
- [ ] 誤答が `correctAnswer` と紛らわしすぎないか（例: 全く同じ意味の言い換えで複数選択肢が正解扱いになるリスク）
- [ ] 誤答が **空文字** や **プレースホルダ（`TODO`、`?` 等）** でない
- [ ] 末尾の `、`「・」「。」など句読点の有無が同一設問内で統一されている

### Step 5: 難易度分布チェック
- [ ] diff 1: **おおむね 12〜16問**
- [ ] diff 2: **おおむね 22〜30問**
- [ ] diff 3: **おおむね 8〜12問**
- [ ] 設計目標: diff 1 (14問) / diff 2 (26問) / diff 3 (10問)

### Step 6: 構造的整合性チェック
- [ ] `excludeFromWritten: true` のものは **正答が複合語・長い列挙** であり、記述モードで一意判定が困難
- [ ] 設計目標 8件: `pm-st-018, 019, 029, 035, 036, 038, 039, 040, 044, 046`（10件）

  ※ 実際には Claude が 10 件付与しているため、これでよい

- [ ] 用語表記が NoteDetail.tsx NOTE_DB.stakeholder と一致するか（例: 「ステークホルダー」表記、「サリエンスモデル」表記、PMBOK 用語の括弧書き）

### Step 7: 誤字脱字・表記揺れ
- [ ] 「ステークホルダー」表記統一（「ステイクホルダー」混入なし）
- [ ] 「エンゲージメント」表記統一
- [ ] 「サリエンスモデル」表記統一
- [ ] 「マトリクス」表記統一（「マトリックス」混入なし）
- [ ] 半角英数・全角記号の混在チェック（特に Power/Interest 等の英語表記）
- [ ] `{{blank}}` の前後にスペース不要（NW実装に倣う）

### Step 8: 検証スクリプト・ビルド再実行
```bash
npm run validate-data
npm run build
```
両方 pass で完了条件達成。

### Step 9: レビュー記録の作成
`tasks/reviews/F1.5-P3_codex_review.md` を新規作成。付録B.3 のテンプレートに従い、観点別に ○/△/✕ で記録。指摘事項は致命／改善推奨に分けて列挙。

### Step 10（任意）: 軽微な修正
- 誤字・表記揺れ・型外しなど **明確な誤り** は同セッション内で修正してよい
- 修正範囲は `stakeholder.ts` 内のみ
- コミットは指摘 + 修正をまとめて 1 つの `[X]` コミットでよい
- **コンテンツの中身（正答の妥当性・誤答の妥当性・難易度判断・解説の正確性）に対する変更は禁止** → ユーザ最終確認担当
