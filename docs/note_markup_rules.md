## 1. マークアップの2系統

PM Learning App のノート本文では、強調表現を 2 系統に分ける。

| 記法 | 表示 | 意味 | マスクトグル |
|---|---|---|:---:|
| `==X==` | **赤字** | RedWord: 暗記対象のキーワード | ✅ 対象 |
| `__X__` | **ネイビー (#9d5b8b)** | NavyWord: 構造的・非暗記対象の強調 | ❌ 対象外 |

## 2. 判断軸

**「試験で答えとして問われる用語か？」** を基準に分ける。

### 2.1 赤字（`==X==`）にすべきもの

- PMBOK／IPA で固有名のある専門用語
  - 例: ステークホルダー登録簿、サリエンスモデル、サーバントリーダーシップ、タックマンモデル、マネジリアル・グリッド
- 分類軸の正式名称
  - 例: 内部／外部、顕在／潜在、Power／Influence／Impact／Interest、X理論／Y理論
- プロセス名・成果物名・ツール&技法の名前
  - 例: 資源マネジメント計画書、チーム憲章、コンフリクト・マネジメント
- 暗記すべき段階・順序・数値
  - 例: 5段階関与レベル、3属性、PMBOK第7版、9.4 チームの育成
- 試験のひっかけ語の正式表記
  - 例: 期待理論（Vroom）、公平理論（Adams）

### 2.2 ネイビー（`__X__`）にすべきもの

- 番号ラベル
  - 例: `__1.__`、`__成果1__`、`__対策1__`、`__S1 指示型__`
- 列挙の頭の役割名・カテゴリ名（本文の方が要点）
  - 例: `__スポンサー__:`、`__顧客__:`、`__機能部門マネジャー__:`
- セクション内のサブ見出し
  - 例: `__午前II 頻出__:`、`__午後Ⅰ 頻出__:`、`__計画書の主な項目__:`
- 一般語の構造的強調
  - 例: `__個人__`／`__組織__`、`__誰に__`／`__何を__`／`__いつ__`、`__重要度__`／`__緊急度__`
- 観点ラベル
  - 例: `__Influence vs Impact__:`、`__Power vs Influence__:`、`__期待理論 vs 公平理論__:`
- 表現的強調
  - 例: `__生きた文書__`、`__単一の正解はない__`、`__役職に依存しない__`

### 2.3 判断に迷ったら

「これを試験で正解として答えるか？」と自問する。**No なら `__X__`（ネイビー）。**

リスト先頭が `==役職名==:` や `==番号==:` の形になる場合は、**ほぼ確実に `__X__` が正解**。

## 3. ❗ よくある不整合パターンと回避法

F2-P1 team 投入時に大量発生した不整合の事例。**生成時から意識して回避する。**

### 3.1 開閉ミスマッチ `__X==`

❌ **悪い例**: `'__変革型リーダーシップ==（Transformational Leadership, Bass）: ==ビジョン==で〜'`
- `__` で開いて `==` で閉じている → マークアップエラー

✅ **良い例**: `'__変革型リーダーシップ__（Transformational Leadership, Bass）: ==ビジョン==で〜'`
- `__` 同士で閉じる

#### 検出コマンド（PowerShell）
```powershell
$content = Get-Content src/pages/NoteDetail.tsx -Encoding UTF8
for ($i = 0; $i -lt $content.Length; $i++) {
  $line = $content[$i]
  $us = ([regex]::Matches($line, '__')).Count
  $eq = ([regex]::Matches($line, '==')).Count
  if (($us -gt 0 -and $us % 2 -ne 0) -or ($eq -gt 0 -and $eq % 2 -ne 0)) {
    "L$($i+1): __[$us] ==[$eq]"
  }
}
```
**完了条件**: 出力が 0 件であること。

### 3.2 番号ラベル + 文章の混同 `__N. <内容>==`

❌ **悪い例**: `'__1. 当事者間==で解決を試みる'`
- 「1. 当事者間」がラベル、「で解決を試みる」が文章のつもりだが、`==` で閉じているため壊れる

✅ **良い例 A**: `'__1.__ ==当事者間==で解決を試みる'`
- 番号だけをネイビーラベルにし、キーワードは赤字で強調

✅ **良い例 B**: `'__1. 当事者間__で解決を試みる'`
- 「1. 当事者間」全体をネイビーラベルとして1つにまとめる

**推奨**: A（番号と内容を分離）。読み手にとって「番号 → ラベル」の構造が明確。

### 3.3 全角イコール `＝` を文中に使わない

❌ **悪い例**: `'X＝怠惰／Y＝自律'`
- 全角イコールはマークアップトークン `==` との混同を招き、レビュー検査時の誤検出原因にもなる

✅ **良い例**: `'Xは怠惰／Yは自律'`、`'X 怠惰 ／ Y 自律'`、または `'X: 怠惰 ／ Y: 自律'`

❌ **悪い例**: `heading: '32. トーマス＝キルマンの5つの対処モード'`
✅ **良い例**: `heading: '32. トーマス-キルマンの5つの対処モード'`（半角ハイフン）

**重要: items / navyItems だけでなく `exam_tips` も対象**。「変更管理＝プロセス、構成管理＝成果物」のような短文対比で混入しやすい（F2-P1 integration §24 教訓）。

### 3.3b ★赤字マークアップ直後に半角イコール `=` を置かない（F2-P1 delivery+measurement 教訓）

❌ **悪い例**: `'==±3σ===99.73%'`
- `==±3σ==` の閉じ `==` の直後に `=99.73%` の `=` が来ると、**3連続 `===`** とパーサーが誤認しやすい
- 検出コマンドで `===` 違反として引っかかる

✅ **良い例**:
- `'==±3σ==は99.73%'`（接続詞「は」で文章化）
- `'==±3σ==: 99.73%'`（コロンで分離）
- `'==±3σ==（99.73%）'`（括弧で分離）

❌ **他の悪い例**:
- `'==6σ===3.4不良/百万'` → `'==6σ==は3.4不良/百万'`
- `'==水平===停滞、==上昇===要警戒'` → `'==水平==は停滞、==上昇==は要警戒'`
- `'==UAT==＝実ユーザの最終受入'` → `'==UAT==は実ユーザの最終受入'`（全角イコールも同様）

**検出パターン**: 行内に `===` または `====` が含まれる場合は要修正。

### 3.4 入れ子・連続記号

❌ **悪い例**:
- `===重要===`（連続3つ）
- `____強調____`（連続4つ）
- `==外側 ==内側== 外側==`（入れ子）

✅ **良い例**:
- `==重要==`
- `__強調__`
- 入れ子は構造上不可。1つの記号ペアで完結させる

### 3.5 navyItems の構造化形式

`navyItems` は `EmphasisToken[]` の配列の配列。`style` は `'navy'` のみ（赤字は items 内の `==X==` で表現する設計）。

✅ **良い例**:
```ts
navyItems: [[{ text: '出典: PMBOK第6版 §9.4', style: 'navy' }]],
```

❌ **悪い例**:
```ts
navyItems: [[{ text: '出典: PMBOK第6版 §9.4', style: 'red' }]],   // style 違反
navyItems: '出典: PMBOK第6版 §9.4',                                // 構造違反
```

### 3.6 navyItems の text 内には `==X==` / `__X__` を書かない ★F2-P0 教訓

`navyItems` は `renderTokens` で描画される。token の `text` フィールド内に `==X==` や `__X__` を書いても、それは **赤字／ネイビーマークアップとして解釈されず、リテラル文字として表示される**。

❌ **悪い例**:
```ts
navyItems: [[{ text: 'ツール&技法に==根本原因分析==・==ステークホルダー分析==を含む', style: 'navy' }]],
```
→ 画面には `ツール&技法に==根本原因分析==・==ステークホルダー分析==を含む` がそのまま表示される。

✅ **良い例**:
```ts
navyItems: [[{ text: 'ツール&技法に根本原因分析・ステークホルダー分析を含む', style: 'navy' }]],
```

`renderText`（`items` 用、`text.split(/==.+?==|__.+?__/g)` でパース）と `renderTokens`（`navyItems` 用、`EmphasisToken[]` を直接描画）で挙動が異なる。

**ルール**: 強調が必要なら、`navyItems` に複数の token を持たせて分割する。
```ts
navyItems: [[
  { text: 'ツール&技法に', style: 'navy' },
  { text: '根本原因分析', style: 'red' },  // navyItems 内の赤字は禁止（§3.5）
  { text: '・', style: 'navy' },
]],
```
ただし `navyItems` 内に `red` style を混ぜることは §3.5 で禁止されているため、結局「`navyItems` の token text は装飾なしの素文字列」が原則。

## 4. 正規表現置換の罠

`__X==` 不整合を一括置換するとき、単純な `__([^_]+?)==` → `__$1__` は **正しい構文も壊す**。

### 4.1 危険なパターン

入力: `'__成果1__: ==共有されたオーナーシップ=='`（正しい構文）
- `__([^_]+?)==` で lazy マッチすると、`__成果1__: ==` 全体をマッチしてしまう
- 置換結果: `'__成果1__: __共有されたオーナーシップ=='`（**壊れる**）

### 4.2 安全な正規表現

`__([^_=]+?)==` を使う（`_` も `=` も含まない文字列のみマッチ）。

```powershell
[regex]::Replace($line, '__([^_=]+?)==', '__$1__')
```

それでも `__X__: __Y==` のような二重不整合は残るので、置換後に再検査して手動で `__X__: ==Y==` に直す。

## 5. インデント（階層構造）の運用 ★必須

ノート本文の `items` (string[]) と `navyItems` (EmphasisToken[][]) は、**包含関係のある項目を全角スペースでインデント表現する**。並列項目を平坦に並べることは禁止。

### 5.1 ルール

- インデントは **全角スペース `　` 1個ずつ** ずらす（最大2階層）
- 親レベル（インデント0）: 行頭に全角スペースなし
- 子レベル（インデント1）: 行頭に `　` 1個
- 孫レベル（インデント2）: 行頭に `　　` 2個

### 5.2 インデントを使うべきケース

- 「親見出し:」の直後に並ぶ列挙項目
  - 例: `主要ツール&技法:` の下にぶら下がる7技法
  - 例: `5つの代表スタイル:` の下にぶら下がる S1-S4
- 番号付きステップが1つの上位概念に従属するとき
  - 例: 「紛争解決の優先順位」の下の `1. 〜` 〜 `6. 〜`
- 分類軸に従属する具体例
  - 例: 「文化次元（Hofstede）:」の下の6次元
- ITTO のような構造（プロセス → ツール&技法 → 各技法）

### 5.3 ❌ 悪い例（平坦）

```ts
items: [
  '5つの代表スタイル:',
  '__1,1 無関心型__: 両方低い',
  '__9,1 仕事中心型__: 業績重視',
  '__1,9 人間中心型__: 人間関係重視',
  '__9,9 チームマネジメント型__: 両方高い',
],
```

### 5.4 ✅ 良い例（階層化）

```ts
items: [
  '5つの代表スタイル:',
  '　__1,1 無関心型__: 両方低い',
  '　__9,1 仕事中心型__: 業績重視',
  '　__1,9 人間中心型__: 人間関係重視',
  '　__9,9 チームマネジメント型__: 両方高い',
],
```

### 5.5 3層構造の例（インデント2 使用）

```ts
items: [
  '__主要ツール&技法__:',
  '　__対人スキル__:',
  '　　==コンフリクト・マネジメント==',
  '　　==意思決定==',
  '　　==感情的知性==',
  '　__プロジェクトマネジメント情報システム__（PMIS）',
],
```

### 5.5b ★子レベル `:` の下の列挙は孫レベルに落とす（F2-P1 integration 教訓）

子レベル（インデント1）の項目が `:` で終わって列挙を続ける場合、列挙は**孫レベル（インデント2）**に落とす。子レベルのままだと「兄弟」として扱われて階層が崩れる。

❌ **悪い例**（整合性違反）:
```ts
items: [
  '__変更要求書==:',
  '　__主要記載項目__:',         // ← 子レベル (1)
  '　==変更ID==／==要求者==',     // ← 子レベル (1)  ← 上の項目と兄弟と誤認
  '　==変更内容==',
],
```

✅ **良い例**:
```ts
items: [
  '__変更要求書==:',
  '　__主要記載項目__:',         // ← 子レベル (1)
  '　　==変更ID==／==要求者==',   // ← 孫レベル (2)  親「主要記載項目」の配下
  '　　==変更内容==',
],
```

**判定基準**: 親項目が `:` で終わる場合、その直下の項目は==必ず一段深く==する。

### 5.6 判定基準

迷ったら次の問いに答える：
- 「この項目は、直前の親項目の **下位要素** か？」 → Yes なら子（インデント1）
- 「この項目は、上の階層の **兄弟要素** か？」 → Yes なら同階層
- 1つの段落で説明が完結する単独項目は親（インデント0）

### 5.7 レンダリング側のサポート

`NoteDetail.tsx` の `items` / `navyItems` 表示分岐は、行頭の全角スペース個数を検出して padding（`ml-5` / `ml-10`）を適用する。レンダリングロジックを変更する場合は、本ルールとの整合性を維持すること。

## 6. レビュー時のチェックリスト

Codex または Claude がレビュー時に必ず実行する項目:

### マークアップ整合性
- [ ] 各 items 文字列について、`__` の出現回数が偶数（または 0）
- [ ] 各 items 文字列について、`==` の出現回数が偶数（または 0）
- [ ] 全角イコール `＝` が含まれない（heading / items / navyItems / **exam_tips** すべて対象、F2-P1 integration 教訓）
- [ ] `===` / `___` / `====` / `____` のような連続記号がない（**赤字直後の `=` で誘発しやすい**、F2-P1 delivery+measurement 教訓）
- [ ] `EmphasisToken.style` が `'red' | 'navy' | 'plain'` 以外を持たない
- [ ] `navyItems` の token は全て `style: 'navy'`
- [ ] **`navyItems` の token text 内に `==X==` / `__X__` が含まれていない**（F2-P0 教訓、§3.6 参照）
- [ ] 各 section の `heading` が「N. タイトル」形式の連番

### インデント整合性（追記）
- [ ] **子レベル `:` の直下の列挙は孫レベル（全角2スペース）に落としている**（F2-P1 integration 教訓、§5.5b 参照）

### インデント整合性 ★v1.0 で追加（必須）
- [ ] 「親見出し:」の直後に並ぶ列挙項目に全角スペースインデントが付いている
- [ ] 番号付きステップ（`__1.__` 〜 `__N.__`）が同一階層で揃っている（並列なら全部親、従属なら全部子）
- [ ] インデント記号は **全角スペース `　`** のみ（半角スペース・タブ・点リーダー等は使わない）
- [ ] インデント深度は最大2階層（孫まで。それ以上深くなる構造はセクション分割を検討）
- [ ] 同一階層の項目は同じインデント深度で揃っている（混在しない）

### 推奨検出コマンド（PowerShell）
```powershell
# 包含関係の疑い: 「:」で終わる項目の直後に同階層・素のテキストが続く箇所をリストアップ
$content = Get-Content src/pages/NoteDetail.tsx -Encoding UTF8
for ($i = 0; $i -lt $content.Length - 1; $i++) {
  if ($content[$i] -match "^\s+':.+:',?$" -and $content[$i+1] -match "^\s+'[^　]") {
    "L$($i+1) → L$($i+2): 親候補（『:』で終わる）の直後にインデントなしの項目"
  }
}
```
