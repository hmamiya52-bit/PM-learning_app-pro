# PM Learning App リスクレジスタ

3視点レビュー（マネージャ目線 M2）で指摘された主要リスクを一覧化。
プロジェクト進行中も随時更新する。

## 🔴 高リスク（早期対応）

### R1: IPA著作権規約違反リスク
- **リスク**: 公式午前II・午後I 過去問題の引用が IPA 規約に違反 → 公開停止要請
- **影響**: アプリ運用停止、ブランドダメージ
- **検知トリガー**: IPA からの問合せ、利用者からの指摘
- **発生確率**: 低（教育目的引用は許諾不要、出典明記必須を守れば問題なし）

#### 調査結果（2026-05-06）
- IPA著作権は放棄していないが、教育目的の引用は許諾・使用料不要
- 出典明記必須: 「出典：<年度> <期> プロジェクトマネージャ試験 午前II 問<番号>」形式
- 改変している場合は明記
- ダウンロード利用は問題なし

#### 対応プラン
- 全公式午前II問題画面で出典をフッタ表示
- `IpaCopyrightFooter` コンポーネント実装（detailed_design §7.4）
- 改変なし運用（解説のみ独自作成、問題文・選択肢はそのまま引用）
- 不安なら IPA 直接問合せで明示確認（必須ではない）

#### 出典フォーマット例
- 年度別: `出典：令和6年度 春期 プロジェクトマネージャ試験 午前II 問3`
- 一覧: `本アプリは独立行政法人情報処理推進機構（IPA）が公開する情報処理技術者試験 過去問題を引用しています。`

### R2: コンテンツ品質保証リスク
- **リスク**: Claudeが生成したノート・解説に誤りがあり、ユーザが間違った知識を学習
- **影響**: 試験合格率低下、アプリ信頼性失墜
- **検知トリガー**: ユーザレビュー、PM試験経験者からの指摘

#### 対応プラン（2026-05-16 体制変更反映: Gemini レビュー廃止）
1. **Codex レビュー**（構造・型・誤字・コードレベル整合性）を必須化（要件定義 §1.6）
2. **パイロット運用フェーズ（1.5）** で1カテゴリ先行検証
3. **ユーザ最終確認**（コンテンツ妥当性・PMBOK整合性・表現の自然さ）を全コンテンツに必須化（Gemini に委ねていた観点を吸収）
4. PMBOK第7版 + IPAシラバスを引用元として明記

---

### R3: PMBOK改訂リスク
- **リスク**: PMBOK第8版（既出）とIPAシラバスの整合性
- **影響**: コンテンツ全面書き直しの可能性
- **検知トリガー**: IPA PM試験シラバス改訂発表

#### 対応プラン
- **メインカテゴリ12個はPMBOK第7版ベースで作成**（基本設計 §5.1.1）
- **第8版の追加・変更要素は専用カテゴリ `pmbok8-diff` で別管理**（基本設計 §5.1.2）
  - 6原理（旧12から再定義）/ 7パフォーマンス領域（旧8から再編）/ プロセス40（49から削減）/ AI付録
- IPA シラバスが第8版を取り込むタイミングを定期ウォッチ（2-3ヶ月毎）
- 取り込み確認後、`pmbok8-diff` を 12カテゴリに統合する判断

## 🟡 中リスク（監視継続）

### R4: LocalStorage 5MB上限超過リスク
- **リスク**: 論述本文＋公式午前II履歴＋ノート理解度等で 5MB 超過 → データ喪失
- **影響**: 学習履歴喪失、ユーザ離脱

#### 試算
- 論述1Attempt 平均6KB × 100件 = 600KB
- 公式午前II問題本文 25問×全年度（約13年）= 約500KB
- ノート本文（投入後）= 約1〜2MB
- 合計 = 約3〜4MB（5MB 内に収まるが余裕は少ない）

#### 対応プラン
- `safeSetItem` で QuotaExceededError をキャッチ
- ユーザに容量超過時にtoast通知 + 古い論述履歴の削除を促す
- 重要マーク・ノート本文は静的データ（バンドル）で LocalStorage 圧迫しない

---

### R5: Vercel無料枠超過リスク
- **リスク**: 月100GB 帯域超過 → デプロイ停止
- **影響**: アプリアクセス不能

#### 試算
- 想定ユーザ数: 個人＋知人数名（10名以下）
- 1ユーザ月間アクセス: 約100MB（PWAインストール後はキャッシュで激減）
- 月間合計: 約1GB → **5%以下、十分な余裕**

#### 対応プラン
- 現状監視のみ。超過の兆候があれば Cloudflare Pages 等への移行検討（未来課題）
- ユーザ確定: **問題ないと判断**

---

### R6: Claude/Codex協業の同時編集衝突リスク
- **リスク**: ターン制ルール違反 → git push 拒否 → 手動rebase必要
- **影響**: 作業時間ロス、最悪コード喪失

#### 対応プラン
- detailed_design §1.4・§1.5 で「同時編集禁止」「pull/push必須」を明文化
- 衝突発生時の rebase手順 を §1.5.7 に記載
- 万一衝突したら `git pull --rebase origin main` で復旧

---

## 🟢 低リスク（受容）

### R7: NW実装の隠れバグ継承
- **リスク**: NWアプリの未発見バグがPMアプリに継承
- **影響**: 同種のバグがPMアプリにも発生

#### 対応プラン
- **F1-P-1 NW実装精読セッション**（v0.14新設）で既知問題を洗い出し
- 重大な箇所はPM側で修正（NWに反映するかは別判断）

---

### R8: 工数超過リスク
- **リスク**: 個人開発で並行プロジェクト多忙 → 計画遅延
- **影響**: フェーズ完了が想定より遅れる

#### 対応プラン
- 工数概算は「最短40日」「現実的2〜4ヶ月」と幅を持たせて記載（detailed_design §1.2）
- 焦らず進める。マイルストーンの日付は柔軟に調整
